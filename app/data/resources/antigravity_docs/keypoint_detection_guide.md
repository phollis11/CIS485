# Keypoint Detection Data Guide

This document explains the structure, meaning, and model logic behind the `keypoints.json` file used in the football field detection pipeline.

## 1. JSON File Structure
The `keypoints.json` file is a serialised list of frame-by-frame pitch landmarks. Each entry in the JSON array corresponds to a single video frame.

```json
[
    {
        "frame": 0,
        "keypoints": [
            [
                [x1, y1, conf1],
                [x2, y2, conf2],
                ...
                [x32, y32, conf32]
            ]
        ]
    }
]
```

### Data Fields:
- **`frame`**: The chronological index of the frame in the video/image sequence.
- **`keypoints`**: A nested array containing detected landmarks.
  - **`x` & `y`**: Pixel coordinates of the point in the source image.
  - **`confidence`**: A value between 0.0 and 1.0. A `0.0` or `[0, 0, 0]` entry indicates the point was not detected or is outside the frame.

---

## 2. Positional Mapping (Keypoint Labels)
The JSON data does **not** contain text labels. Instead, it uses a **fixed positional mapping** where the index of the point determines its identity. This 32-point layout follows the **Roboflow Football Pitch Standard**.

| Index | Landmark Name | Region |
| :--- | :--- | :--- |
| **0** | Top-Left Corner | Boundary |
| **1** | Bottom-Left Corner | Boundary |
| **2-5** | Left Penalty Area Corners | Left Side |
| **6-9** | Left Goal Area Corners | Left Side |
| **10** | Left Penalty Spot | Left Side |
| **11-13** | Left Penalty Arc | Left Side |
| **14** | Top Midline Point | Center |
| **15** | Center Spot | Center |
| **16** | Bottom Midline Point | Center |
| **17-23** | Center Circle Arcs | Center |
| **24** | Top-Right Corner | Boundary |
| **25** | Bottom-Right Corner | Boundary |
| **26-29** | Right Penalty Area Corners | Right Side |
| **30-31** | Right Goal Area Corners | Right Side |

---

## 3. Relationship to Keypoint Detection Models (YOLO-Pose)
This data is generated by a **Pose Estimation Model** (like YOLOv8-pose or YOLO11-pose).

### How the Model "Thinks":
- **Fixed Output Tensor**: The model is trained to always predict exactly 32 points in the same order. It does not "recognize" a penalty spot by name; it simply learns that the 10th set of coordinates it outputs should align with that visual feature.
- **Confidence Scores**: The third value in each triplet (`conf`) is the model's self-assessment. In your pipeline, points with low confidence are usually ignored when drawing the field lines to prevent "jitter."
- **Symmetry**: The model uses the spatial relationship between points to predict "occluded" or off-screen points. For example, if it sees the midline and one penalty area, it can often guess where the other corners are.

---

## 4. Practical Application
### Drawing (The Pitch Skeleton)
To visualize the field, the script connects specific indices with lines (defined in `PITCH_SKELETON`).
*Example:* Connecting point 4 (Left Penalty Top-Right) to point 2 (Left Penalty Top-Left).

### Homography (Mapping to TOP-DOWN View)
By selecting at least 4 known points (e.g., indices 0, 1, 24, 25) and comparing their pixel coordinates to actual real-world dimensions (e.g., 105m x 68m), you can calculate a **Homography Matrix**. This allows you to transform every pixel in the video into a 2D coordinate on a tactical map.

> [!IMPORTANT]
> When processing this JSON, always check if `x` and `y` are greater than 0. If they are exactly 0, the model has failed to locate that specific landmark in that frame.
